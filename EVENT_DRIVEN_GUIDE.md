## Часть 1. Базовые понятия (читать первой)

### Что такое «сервис» (микросервис)

Наш проект — это не одна программа, а **несколько маленьких программ**, каждая
со своей задачей. Они работают одновременно и общаются между собой. Каждую
такую программу называют **сервисом**.

У нас их пять (плюс вспомогательные):

| Сервис | Простыми словами, что делает |
|--------|------------------------------|
| **api-gateway** | «Администратор на ресепшене». Единственный, к кому обращается браузер. Проверяет, кто ты, и раздаёт задания остальным. |
| **file-service** | «Кладовщик». Принимает файлы и кладёт их в хранилище (S3/MinIO). |
| **extraction** | «Чтец». Читает документ (PDF/Word) и через нейросеть (LLM) достаёт из него характеристики. |
| **domain-analyze** | «Сравниватель». Берёт характеристики из ТЗ и из паспорта и сравнивает их (тоже через LLM). |
| **prompt-registry** | «Шпаргалочница». Хранит тексты заданий (промптов) для нейросети. |
| **knowledge-base** | «Справочник». Хранит знания (синонимы, единицы измерения), помогает нейросети не ошибаться. |

### Два способа, которыми сервисы общаются

Это **самое важное** для созвона. Запомни эти две схемы.

**Способ 1 — синхронный вызов (HTTP-запрос).**
Один сервис звонит другому и **ждёт ответа прямо сейчас**, как телефонный звонок:

```
api-gateway: «file-service, дай мне ссылку на файл»
file-service: «вот, держи» (отвечает сразу)
api-gateway: ... ждал, получил ответ, работает дальше
```

В коде это `httpx` — библиотека для HTTP-запросов. Минус: пока ждёшь ответ,
ты «висишь». Если собеседник медленный (например, нейросеть отвечает 2 минуты) —
ты 2 минуты ничего не делаешь.

**Способ 2 — асинхронная задача через очередь (Celery + RabbitMQ).**
Один сервис **кладёт задание в очередь** и сразу идёт дальше, как записка на доске:

```
api-gateway: пишет на доску «нужно извлечь данные из файла X» и уходит
              (не ждёт!)
worker:      когда освободится — берёт записку с доски и делает работу
worker:      закончил → стучится обратно: POST /callback «готово, вот результат»
```

- **Очередь** (queue) — это «доска с записками». У нас её роль выполняет
  **RabbitMQ**.
- **Worker** — отдельная программа того же сервиса, которая разбирает доску.
  В нашем `docker-compose.yml` у сервисов есть две роли: обычный сервер
  (отвечает на HTTP) и worker (`celery ... worker`).
- **Callback** («обратный звонок») — когда worker закончил, он сам стучится
  обратно по HTTP и говорит «готово».

> 💡 Главное: **очередь нужна для долгих задач** (скачать файл, спросить
> нейросеть). Чтобы пользователь не ждал у экрана 2 минуты, мы говорим ему
> «принято, обрабатываю» и делаем работу в фоне.

### Что такое «событие» (event) и event-driven стиль

Теперь — про что будет созвон.

**Событие** — это сообщение «случился факт» в прошедшем времени:
- `ФайлЗагружен`
- `ХарактеристикиИзвлечены`
- `СравнениеЗавершено`

**Event-driven** («управляемый событиями») — это стиль, где сервисы общаются
**только через события**. Сервис не приказывает другому «сделай Х». Он просто
объявляет на весь мир: «у меня случилось Х». А другие сервисы сами решают,
интересно им это или нет, и реагируют.

Сравни два стиля на примере «как организовать работу»:

**Оркестрация** (как у нас сейчас) — есть **дирижёр**, который всем управляет:
```
Дирижёр (api-gateway):
  - «file-service, загрузи файл»
  - получил ответ → «extraction, извлеки данные»
  - получил ответ → «domain-analyze, сравни»
  - получил ответ → «готово, показываю пользователю»
Дирижёр знает ВЕСЬ процесс и держит всю логику у себя.
```

**Хореография** (куда зовёт event-driven) — **дирижёра нет**, каждый танцор
знает свою партию и реагирует на музыку:
```
file-service:    «Я загрузил файл!» (публикует событие ФайлЗагружен)
extraction:      услышал ФайлЗагружен → извлёк → «Я извлёк данные!»
domain-analyze:  услышал событие про паспорт → сравнил → «Я сравнил!»
api-gateway:     слушает всё и просто обновляет экран пользователя.
Никто не командует — все реагируют на события.
```

### Зачем вообще event-driven? (плюсы и минусы — честно)

**Плюсы:**
- Сервисы не знают друг о друге → легче добавлять новые. Захотел добавить
  «сервис уведомлений» — он просто подписывается на нужные события, и менять
  старый код не надо.
- Долгие и нестабильные операции (нейросеть, поиск) легче делать надёжно:
  каждый шаг — отдельное событие, упавший шаг можно повторить отдельно.
- Нагрузку легче масштабировать: много событий → запусти больше worker'ов.

**Минусы (и это честные вопросы для созвона):**
- **Тяжелее отлаживать.** Когда нет дирижёра, логика «что после чего»
  размазана по всем сервисам. «Почему анализ застрял?» — найти ответ сложнее.
- Нужны дополнительные инструменты: трассировка (см. ниже), схемы событий.
- Для маленького проекта это бывает **избыточно**. Оркестрация проще.

> 📌 **Твоя позиция на созвоне:** у нас уже есть половина event-driven
> (очереди, callback'и), но логика собрана в дирижёре (`api-gateway`). Разговор,
> скорее всего, про то, стоит ли двигаться в сторону хореографии — и где это
> реально поможет, а где усложнит.

## Часть 2. Полный путь документа — по шагам и номерам строк

Здесь — **сквозной маршрут**: что происходит, когда пользователь входит,
загружает ТЗ и паспорт, и в конце видит таблицу сравнения. Каждый шаг помечен
**файлом и номером строки**, чтобы ты мог открыть и посмотреть своими глазами.

> Номера строк актуальны на момент написания (после рефакторинга). Если код
> поменяется — числа могут сдвинуться, но имена функций останутся ориентиром.
>
> **Фронтенд** (кнопки в браузере) живёт в отдельном проекте `pdf-analyzer` —
> здесь я указываю, **какой запрос к бэкенду** отправляет каждая кнопка.

### Шаг 0. Пользователь входит в систему (авторизация)

Пользователь вводит логин/пароль и жмёт «Войти». Браузер шлёт `POST /auth/login`.

| Что происходит | Файл : строка |
|----------------|---------------|
| Принимается запрос на вход | `services/api-gateway/app/api/auth.py:200` — функция `login_user` |
| Проверяется пароль (argon2) | `auth.py:209` (`verify_password`) |
| Создаётся сессия + CSRF-токен | `auth.py:212` → функция `create_user_session` на `auth.py:100` |
| В браузер ставятся cookie (сессия + защита от подделки) | `auth.py:129` и `auth.py:130` (`set_cookie`) |

После этого **каждый** следующий запрос проходит «охранника» — middleware:

| Что происходит | Файл : строка |
|----------------|---------------|
| Охранник перехватывает любой запрос | `services/api-gateway/app/main.py:45` — функция `auth_gate` |
| Узнаёт пользователя по cookie | `main.py:51` (`get_current_user_optional`) |
| Проверяет CSRF-токен для «опасных» методов (POST/PUT/...) | `main.py:59` (`validate_csrf`) |
| Если не узнал — отвечает 401 «не авторизован» | `main.py:66` |

> 🧠 Это **синхронный** шаг (Способ 1): пользователь ждёт ответа сразу.
> Здесь нет событий — авторизация быстрая, очередь не нужна.

### Шаг 1. Пользователь загружает ТЗ и паспорт

Пользователь выбирает два файла и жмёт «Загрузить». Браузер шлёт
`POST /files/upload` (с файлами внутри).

| Что происходит | Файл : строка |
|----------------|---------------|
| Принимается загрузка двух файлов | `services/api-gateway/app/api/files.py:25` — `upload_files` |
| Создаётся запись `Analysis` со статусом `processing_files` | `files.py:38` (внутри `upload_files`, `Analysis(... status="processing_files")`) |
| Файлы пересылаются «кладовщику» | `files.py:89` — POST в `file-service /files/upload-batch` |

Дальше работает **file-service** (кладовщик):

| Что происходит | Файл : строка |
|----------------|---------------|
| Принимает файлы | `services/file-service/app/api/routes.py:74` — `upload_batch` |
| **Кладёт задачу в очередь** (Способ 2!) — по одной на каждый файл | `routes.py:98` и `routes.py:101` (`upload_to_s3.delay(...)`) |
| Worker заливает файл в S3/MinIO | `services/file-service/app/tasks.py:11` — `upload_to_s3` |
| Worker **стучится обратно** (callback): «файл загружен» | `tasks.py:44` — POST в `api-gateway /files/callback` |

> 🧠 Вот первое **«событие в зачаточном виде»**: file-service не приказывает
> ничего, он сообщает факт «файл загружен» через callback. В настоящем
> event-driven это было бы событие `FileUploaded` в общей шине, а не прямой
> HTTP-стук в один конкретный сервис.

### Шаг 2. Система понимает, что оба файла готовы, и запускает извлечение ТЗ

Callback приходит в api-gateway — по разу на каждый файл.

| Что происходит | Файл : строка |
|----------------|---------------|
| Принимается callback «файл загружен» | `services/api-gateway/app/api/files.py:95` — `files_callback` |
| Проверка: **оба** файла уже загружены? | `files.py:149`–`150` (считает `uploaded_count >= 2`) |
| Если да — статус `Analysis` → `extracting_data` | `files.py:154` |
| Создаётся задача `ExtractionJob` для ТЗ и кладётся в очередь | `files.py:193` (`extract_file.apply_async(...)`) |

> 🧠 Здесь хорошо видна **оркестрация**: api-gateway сам РЕШАЕТ, что «раз оба
> файла готовы — пора извлекать». Эта логика-«если» (`uploaded_count >= 2`)
> зашита в дирижёре. В event-driven стиле это решение принимал бы отдельный
> подписчик, а не центральный сервис.

### Шаг 3. Worker извлекает данные из ТЗ через нейросеть

Задачу из очереди берёт worker api-gateway.

| Что происходит | Файл : строка |
|----------------|---------------|
| Worker запускает извлечение | `services/api-gateway/app/tasks.py:80` — задача `extract_file` |
| Вызывает основную логику извлечения | `tasks.py:95` → `run_extraction_task` |
| Берёт текст задания (промпт) у «шпаргалочницы» | `services/api-gateway/app/services/extraction_tasks.py:270` (GET в `prompt-registry`) |
| Отправляет файл «чтецу» на извлечение | `extraction_tasks.py:290` (POST в `extraction /extract`) |

Внутри сервиса **extraction** (самое «дорогое» место — тут работает нейросеть):

| Что происходит | Файл : строка |
|----------------|---------------|
| Принимается запрос на извлечение | `extraction/app/main.py:2262` — `extract_document` |
| Выбирается «движок» (по умолчанию OpenRouter) | `extraction/app/main.py:105` (`_select_backend`) |
| Основная работа: скачать файл, прочитать, спросить нейросеть | `extraction/app/main.py:2002` (`_extract_via_openrouter`) |

Когда ТЗ извлечено, worker api-gateway меняет статус:

| Что происходит | Файл : строка |
|----------------|---------------|
| Статус `Analysis` → `tz_review` (нужна проверка человеком) | `services/api-gateway/app/tasks.py:138`–`142` (ветка `if file_type == "tz"`) |

> 🧠 Этот шаг — **главный кандидат на event-driven улучшение**. Сейчас задача
> `extract_file` делает ВСЁ сразу (скачать + нейросеть + сохранить + решить, что
> дальше). Если нейросеть упадёт — повторяется **вся** задача целиком, включая
> повторную (платную!) попытку. В event-driven каждый под-шаг был бы отдельным
> событием, и повторять можно было бы только упавший кусок.

### Шаг 4. Пользователь проверяет характеристики ТЗ (единственный ручной шаг)

Система показывает пользователю извлечённые характеристики ТЗ. Он отмечает,
какие верные, и жмёт «Продолжить».

| Что происходит | Файл : строка |
|----------------|---------------|
| Браузер запрашивает характеристики для проверки | `services/api-gateway/app/api/analyses.py:603` — `get_tz_review` |
| Пользователь сохраняет правки | `analyses.py:628` — `save_tz_review` (PUT) |
| Пользователь жмёт «Продолжить» | `analyses.py:647` — `continue_tz_review` |
| Статус → `extracting_passport` | `analyses.py:695` |
| Ставится задача извлечь **паспорт** (в очередь) | `analyses.py:760` (`extract_file.apply_async`) |

> 🧠 Это **единственное место**, где процесс ждёт человека. Всё остальное —
> автоматическая цепочка.

### Шаг 5. Извлечение паспорта и запуск сравнения

Worker извлекает паспорт (так же, как ТЗ в шаге 3). Когда паспорт готов:

| Что происходит | Файл : строка |
|----------------|---------------|

| Статус → `analyzing_data` | `services/api-gateway/app/tasks.py:172` |
| Отправляет данные «сравнивателю» | `services/api-gateway/app/tasks.py:186` (POST в `domain-analyze /compare/jobs`) |

Внутри **domain-analyze** (сравниватель):

| Что происходит | Файл : строка |
|----------------|---------------|
| Принимает запрос на сравнение, кладёт в очередь | `services/domain-analyze/app/api/routes.py:21` → `routes.py:25` (`compare_documents.apply_async`) |
| Worker сравнивает ТЗ и паспорт через нейросеть | `services/domain-analyze/app/tasks.py:21` — `compare_documents`, строка `29` (`compare_json`) |
| Worker **стучится обратно**: «сравнение готово» | `services/domain-analyze/app/tasks.py:61` (POST в `api-gateway /compare/callback`) |

### Шаг 6. Сборка результирующей таблицы

Callback о завершении сравнения приходит в api-gateway.

| Что происходит | Файл : строка |
|----------------|---------------|
| Принимается callback «сравнение готово» | `services/api-gateway/app/api/compare.py:17` — `compare_callback` |

Теперь пользователь открывает результат, и браузер запрашивает данные для экрана:

| Что происходит | Файл : строка |
|----------------|---------------|
| Отдаётся таблица + цитаты + подсветка для PDF | `services/api-gateway/app/api/analyses.py:842` — `get_viewer_context` |

И пользователь видит финальную **таблицу сравнения** с подсветкой цитат в PDF. 🎉

### «Светофор» статусов анализа

В таблице `Analysis` есть поле `status` — это как трекинг посылки. По нему
всегда видно, на каком шаге застрял анализ:

```
processing_files  → файлы заливаются
extracting_data   → нейросеть читает ТЗ
tz_review         → ждём проверки человеком
extracting_passport → нейросеть читает паспорт
analyzing_data    → нейросеть сравнивает
ready             → готово, таблица собрана
failed            → где-то ошибка
```

## Часть 3. Где наш проект уже event-driven, а где нет

Чтобы на созвоне говорить предметно.

**Уже есть (хорошо):**
- Очередь сообщений (RabbitMQ) и фоновые worker'ы (Celery).
- Долгие операции (заливка, нейросеть, сравнение) вынесены в фон — пользователь
  не ждёт у экрана.
- Callback'и («стук обратно») — это зачаток событий.
- Авто-повтор упавших задач (`autoretry_for`, backoff) — `tasks.py`.

**Чего не хватает до «настоящего» event-driven (темы для созвона):**
1. **Нет общей шины событий.** Callback'и идут напрямую «сервис → конкретный
   сервис», а не «опубликовал событие — кто хочет, тот слушает».
2. **Логика собрана в дирижёре.** Решения «что после чего» зашиты в api-gateway
   (например, `uploaded_count >= 2` в `files.py:149`). В хореографии их приняли
   бы отдельные подписчики.
3. **Толстые задачи.** `extract_file` делает скачивание + нейросеть + сохранение
   одним куском → при сбое повторяется всё, включая дорогой вызов LLM.
4. **Нет «outbox».** Запись в БД и «стук» в следующий сервис — два отдельных
   действия. Если упасть между ними, состояние рассинхронится.
5. **События без схемы.** Callback'и принимают сырой `dict` (например,
   `compare_callback(payload: dict)`), без описанного контракта.

## Часть 4. Словарик терминов (шпаргалка на созвон)

| Термин | Простыми словами |
|--------|------------------|
| **Событие (event)** | Сообщение «случился факт» (`ФайлЗагружен`). В прошедшем времени. |
| **Оркестрация** | Один «дирижёр» командует всеми (как у нас сейчас). |
| **Хореография** | Дирижёра нет, каждый реагирует на события сам. |
| **Очередь (queue)** | «Доска с записками» для фоновых задач. У нас — RabbitMQ. |
| **Worker** | Программа, которая разбирает очередь и делает фоновую работу. |
| **Callback** | «Стук обратно»: закончил задачу — сообщил инициатору. |
| **Идемпотентность** | Свойство «повторить безопасно»: повтор задачи не делает двойную работу (и двойную оплату LLM). |
| **Outbox pattern** | Приём, чтобы «записать в БД» и «отправить событие» происходили атомарно (либо оба, либо ни одного). |
| **Circuit breaker** | «Предохранитель»: если внешний API (LLM) падает — перестать его долбить на время. |
| **Dead-letter queue** | «Отстойник» для задач, которые не удалось выполнить за N попыток. |
| **Трассировка (tracing)** | Сквозной `id`, который проходит через все события одного запроса — чтобы можно было отследить путь при отладке. |
| **Backpressure** | Защита от перегрузки: если задач больше, чем можем переварить — притормозить приём. |

## Часть 5. Вопросы, которые стоит задать на созвоне

Чтобы созвон был диалогом, а не лекцией (и чтобы ты выглядел подготовленным):

1. «У нас сейчас оркестрация через api-gateway и Celery. С какого момента
   (по нагрузке/масштабу) этого перестаёт хватать и нужна шина событий?»
2. «Наш `extract_file` делает скачивание + вызов LLM + сохранение одной задачей.
   Как правильно разбить это, чтобы при сбое не повторять дорогой вызов нейросети?»
3. «Как обеспечить идемпотентность, чтобы повтор задачи не приводил к двойному
   платному вызову LLM?»
4. «Как решается атомарность “записал в БД + опубликовал событие” (outbox)?»
5. «Как ты трассируешь один бизнес-поток через много событий при отладке?»
6. «Что используешь для медленных/нестабильных внешних API: circuit breaker,
   отдельная очередь под LLM, таймауты?»
7. «С какого одного куска ты бы начал переписывать именно наш проект, чтобы
   получить максимум пользы при минимуме риска?»

> 💬 Не бойся сказать «я пока новичок и разбираюсь». Хорошие вопросы из этого
> списка покажут, что ты подготовился и понимаешь суть, даже если деталей ещё
> не знаешь.
